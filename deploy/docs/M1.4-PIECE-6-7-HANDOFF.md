### Why this is multi-hour work, not 30-minute work

The 19-dep extraction is a fan-out problem. Each dep needs to be:
1. Verified as actually used by the handlers (vs only present in scope by accident)
2. Verified as available at mount time (some are init'd async in server.js)
3. Verified as NOT mutated by the handlers in a way that would lose state across the module boundary

Items 2 and 3 are the dangerous ones. `broker` is the global singleton that gets reassigned during reauth. If the mount function captures it by value, post-reauth orders use the old reference. Either pass a getter, or move the capture into the function body.

Plan-mode breakdown should split this into 3 sub-commits:
- **6a**: Stub `routes/orders.js` with one trivial extracted route (e.g. dry-run, no live-money path). Update one test. Verify.
- **6b**: Move place + cancel + confirm-2fa. Update both test files. Verify.
- **6c**: Remove the cancel-2fa handler from server.js (already moved as a sub-handler in 6b).

## Piece 7 — `routes/ws.js + services/tick-fanout.js`

### Scope

The WebSocketServer block at server.js L4900 to ~L5210 (~310 lines) plus its 7 cross-file dependencies on `wsClients` outside the block.

### Shared mutable state — THE central problem

`wsClients` (Set<WebSocket>) is referenced at:

| Location | Use |
|---|---|
| L4932 (declaration) | `const wsClients = new Set();` |
| L4952 (tick callback) | `for (const ws of wsClients) { ws.send(payload) }` — the hot fan-out loop |
| L4982 (broadcaster #1) | Same iteration pattern for upstream-state changes |
| L4987 (log) | `console.log('[ws] upstream_state changed:', cur, '->', wsClients.size, 'clients');` |
| L1378, L1402 (metrics) | `wsClients.size` for the Prometheus gauge |
| L1457 (alerts broadcast) | `for (const ws of wsClients) { send alert }` |
| L3217 (kill-switch broadcast) | Same pattern |
| L5043 (connection handler) | `wsClients.add(ws)` |
| L5045 (cap check) | `if (wsClients.size > MAX_WS_CLIENTS) ...` |
| L5202 (disconnect handler) | `wsClients.delete(ws)` |

**If `wsClients` moves to a new module, 5 of those references stay in server.js** (the metrics gauge, alerts broadcast, kill-switch broadcast). Either:
- The new tick-fanout module exposes `getClients()` for server.js to iterate, OR
- The Set stays in server.js and ONLY the tick callback + broadcaster intervals move

Option B is the safer one. The new module would be a `services/tick-fanout.js` that exports `attachUpstreamFanout({ broker, wsClients, paper, alerts })` and runs the broadcast loop, but `wsClients` remains a server.js module-level variable. Less elegant but lower risk.

### The WS upgrade handler is the SECOND piece

Separately, `wss.on('connection', ...)` at L5043 is ~150 lines including:
- Cookie-based auth (calls `readSessionCookie`, the legacy ats.sid path)
- T-130 welcome packet construction
- subscribe/unsubscribe message handling
- The per-client `ws.symbolSet`

This handler depends on: `auth`, `readSessionCookie`, `db`, `broker`, `DEFAULT_SYMBOLS`, `MAX_WS_CLIENTS`, plus the `wsClients` Set we just discussed.

### Why this is the hardest extract

The audit explicitly flagged this as the "save for last" extract because:
1. **Real-time hot path**: a bug delays or drops every connected user's ticks.
2. **Shared state across multiple consumers**: tick fanout, alerts broadcast, kill-switch broadcast, metrics, connection-handler all read/write `wsClients`.
3. **Upstream subscription lifecycle**: `broker.subscribeTicks` is invoked once at boot. The fanout callback is captured by reference. Re-attaching across modules is error-prone.
4. **The verifyClient gate** (T-198b) lives in the constructor options — needs to move WITH the wss instantiation or stay where the WSS is created.

### Plan-mode breakdown for piece 7

- **7a**: Extract `services/tick-fanout.js` exporting `attachUpstreamFanout({ broker, wsClients, getMetrics })`. server.js keeps `wsClients` + `MAX_WS_CLIENTS` + DEFAULT_SYMBOLS. Only the upstream subscription + the tick `.then` callback move. Test by inspecting the welcome packet shape via the existing `ws-welcome.spec.js`.

- **7b**: Extract `routes/ws.js` exporting `mountWs(server, { auth, db, broker, wsClients, readSessionCookie, ... })`. The WebSocketServer ctor + the `wss.on('connection', ...)` handler move. The `verifyClient` Origin check (T-198b) moves with the ctor. Test by running the full WS suite.

Both sub-commits need the WS spec suite green (ws-welcome.spec.js + happy-path.spec.js's WS section). The T-198d-restored specs serve as the regression net.
